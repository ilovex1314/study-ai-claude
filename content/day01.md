verifiableOutput: "一份能力边界决策表，列明用户结果、成功指标、模型职责、确定性控制点、高风险动作与人工确认路径。"

walkthrough: "电商退款助手接到用户咨询，系统先识别任务类型（咨询退款资格），再由模型提取订单号并生成解释草稿；订单服务查询资格，规则引擎计算金额，风控策略判断是否要人工确认，最终状态变更由后端服务完成并写入审计日志；模型全程不直接修改业务状态，任何高风险动作都经过 schema 校验与权限过滤才能执行。"
modules:
  - id: "d01-m1"
    title: "先定义用户结果，再决定要不要用 AI"
    concept: "capability"
    idea: "\"接入 AI\"不是任务；任务必须同时说明用户结果、可用证据、错误代价和成功指标。"
    whyItMatters: "没有可测任务，团队会把聊天界面、模型调用次数或主观好评误当成产品结果，最终既无法排错也无法迭代。"
    engineerLens: "先写任务边界卡：用户结果（用户完成什么）、证据（哪些事实支持决策）、系统控制（哪些不能交给模型）、失败兜底和验收指标（量化，可观测）。"
    productionExample:
      context: "某 B2B SaaS 的客服工单系统，日均 5000 条工单，其中 30% 是退款咨询。"
      whatTheyDid: "团队没有先跑模型，而是先写出任务边界卡：用户结果是\"理解退款资格并获得正确下一步\"，成功指标是首次解决率 ≥85% 和人工升级率 ≤15%，不能交给模型的是金额计算、订单状态写入和资质核查。"
      outcome: "边界卡清晰后，团队两周内完成从模型接入到灰度上线，首次解决率达到 87%，误操作率为零。"
    counterExample:
      context: "某内部知识库问答 demo，一周内部署到生产。"
      antiPattern: "直接把\"AI 回答\"当成产品目标，没有定义引用覆盖率、拒答率和版本过滤，上线后政策更新引发多次错误答复。"
      consequence: "两个月内收到 40 条投诉，最终下线重建，损失 3 个月工程资源。"
    pitfalls:
      - symptom: "任务描述只有\"做一个 AI 助手\"，没有用户结果"
        fix: "补充\"用户能完成什么\"、\"什么证据证明成功\"和\"错误的代价是什么\"三个问题的答案"
      - symptom: "成功指标是\"感觉更好用\""
        fix: "改为可观测指标，如首次解决率、引用覆盖率或人工接管率"
  - id: "d01-m2"
    title: "LLM 是概率决策器，不是后端函数"
    concept: "uncertainty"
    idea: "LLM 根据上下文预测 token 分布再采样生成，同一输入不必然得到同样输出；它适合语义判断与候选生成，不适合决定权限、金额和不可逆动作。"
    whyItMatters: "把模型当函数会导致把权限、金额和状态变更交给不稳定文本，轻则数据不一致，重则资损与合规违规。"
    engineerLens: "把 LLM 当语义判断和生成组件；状态机、权限、审计、重试、人工确认留给后端；每个模型输出都要定义 schema、字段含义、空值语义与失败处理。"
    productionExample:
      context: "某金融科技公司的贷款审核辅助系统，日均处理 8000 份申请。"
      whatTheyDid: "模型只负责从申请材料中提取结构化字段（姓名、收入、负债比），输出严格 schema，风控规则引擎用提取结果做信用评分和额度计算，人工审核员最终签批，所有决策都带审计时间戳。"
      outcome: "材料提取准确率从人工的 91% 提升至 96%，但额度和资质判断零变化，完全由规则引擎负责，合规审查一次通过。"
    counterExample:
      context: "某电商平台的自动退款机器人。"
      antiPattern: "让模型判断置信度高于 0.9 就直接调用退款 API，没有幂等键和权限校验。"
      consequence: "一次模型输出格式变更导致 1200 笔重复退款，资损金额超百万，事后无审计记录可追溯。"
    pitfalls:
      - symptom: "模型输出直接写入数据库或触发外部 API"
        fix: "所有高风险动作必须经过系统侧 schema 校验、权限检查和幂等键保护"
      - symptom: "用模型\"置信度\"替代业务规则判断"
        fix: "置信度可作为路由信号（转人工），不能替代授权、余额和合规检查"
  - id: "d01-m3"
    title: "Temperature 控制发散，不控制事实正确"
    concept: "deterministic-fallback"
    idea: "temperature 调整采样随机性：低温减少发散，适合分类与抽取；高温增加多样性，适合候选生成；但它无法让模型\"知道\"不知道的事实。"
    whyItMatters: "团队常把 temperature=0 当可靠性开关，忽略了事实错误主要来自训练截止、知识缺口和上下文噪声，而非采样随机性。"
    engineerLens: "为不同任务设定 temperature 范围（分类/抽取 0~0.2，客服回复 0.3~0.6，方案发散 0.7~1.0）；同时建立事实校验机制：引用校验、工具查询和结构化输出 schema，而非仅靠调参。"
    productionExample:
      context: "某法律科技公司的合同条款抽取服务，每月处理 20 万份合同。"
      whatTheyDid: "抽取任务固定 temperature=0 并要求 JSON 输出；同时在后处理阶段用规则引擎校验必填字段、日期格式和金额单位，抽取错误率过高时降级到人工队列而非重新调参。"
      outcome: "抽取准确率稳定在 94% 以上，降级率低于 3%，规则校验拦截了 78% 的格式错误。"
    counterExample:
      context: "某内部技术问答机器人，用于回答 API 文档问题。"
      antiPattern: "不提供检索上下文，仅将 temperature 设为 0，认为\"低温就不会乱说\"。"
      consequence: "模型用训练记忆回答已废弃 API，导致工程师花费大量时间排查由 AI 建议引起的 bug。"
    pitfalls:
      - symptom: "以为 temperature=0 保证了答案正确"
        fix: "补充检索、引用校验和工具查询；temperature 只控制发散程度"
      - symptom: "创意任务也用极低 temperature，导致输出单调重复"
        fix: "评估任务性质后分配温度范围，发散任务需要更高温度"
  - id: "d01-m4"
    title: "评估是 AI 产品的持续健康指标"
    concept: "eval"
    idea: "没有评估框架的 AI 功能只有上线那天是\"好的\"；Golden dataset、离线回归和在线反馈共同构成迭代闭环。"
    whyItMatters: "模型、prompt 或检索任何一处变更都可能引发退步，只有 eval 闭环才能在发版前发现问题，而不是靠用户投诉。"
    engineerLens: "为每个 AI 功能维护三层评估：离线 golden set（≥20 条覆盖边界场景）、上线 A/B 和影子模式（实验驱动改进）、在线失败回流（用户标记或系统自动归档异常 run）。"
    productionExample:
      context: "某内容平台的自动摘要功能，每天生成 10 万条摘要。"
      whatTheyDid: "建立 500 条 golden dataset，覆盖长文、短文、包含数字/引用/列表的边界情况；每次 prompt 或模型变更都跑离线回归，评分下降超过 2% 则阻断发布；在线按 5% 采样用 LLM-as-judge 自动打分，失败样本每周人工复核一次。"
      outcome: "六个月内发版 14 次，拦截 3 次质量退步，摘要用户满意度从 72% 提升至 84%。"
    counterExample:
      context: "某企业内部的邮件分类助手，没有 eval 体系。"
      antiPattern: "每次改 prompt 都靠工程师主观感受判断\"是否变好\"，没有样例集和指标。"
      consequence: "一次 prompt 重构后分类准确率下降 12%，两周后用户集中反馈才发现，修复成本是发版时评估成本的 20 倍。"
    pitfalls:
      - symptom: "只看单次 demo 效果来判断功能质量"
        fix: "建立至少 20 条覆盖边界场景的 golden dataset，并在每次变更后自动回归"
      - symptom: "Eval 只在上线前跑一次，上线后无监控"
        fix: "在线采样打分和失败回流是持续运营的基础，不是一次性工作"

    name: "上下文策略"
    question: "本次请求需要哪些信息进入上下文？"
    choices:

        description: "当前任务输入、经权限过滤的相关证据片段和本轮工具返回结果，共同组成本次请求的工作台。"
        example: "当前工单内容、用户最近 3 笔订单状态和退款政策摘要。"
      - name: "持久化记忆"
        description: "项目约定、用户长期偏好、历史结论和知识库索引，必须持久化存储并按需检索，不靠上下文维持。"

  - id: "dl01-3"
    name: "可靠性保障"
    question: "模型输出不稳定时如何兜底？"
    choices:
      - name: "结构化输出加校验"
        description: "用 schema 约束模型结果格式，解析失败则触发重试或降级，而不是把原始文本直接传递给下游。"
        example: "要求模型返回 JSON，字段含义和空值语义有明确定义，解析错误则进入人工队列。"
      - name: "评估与人工确认兜底"
        description: "低置信度或高风险场景转人工审核，并把失败样本归档到 eval dataset，驱动下次迭代。"
        example: "退款金额超过 1000 元或模型输出 confidence 低于 0.7 时，自动转人工确认。"
