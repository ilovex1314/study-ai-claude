capabilityGoal: "能交付一份包含架构图、数据契约、eval 计划和复盘模板的 capstone 设计包，让研发和产品可直接基于它实现与验收。"

mentalModel: "Capstone 不是大而全的系统，而是小而完整的交付证据——每个环节都能被验证、复现和复盘。"
walkthrough: "团队选择\"PRD 评审 Agent\"为 capstone。Step1 写 problem statement（谁在痛苦：研发审 PRD 平均 2 小时；非目标：不做全文生成）；Step2 画系统架构图（前端上传区、编排层、检索历史规范库、生成审查清单、trace/log）；Step3 写接口契约（input：文档 URL + meta；output：risks[] + checklist[] + citations[]；error：超时和权限不足）；Step4 写 UX 状态机（idle/uploading/processing/result/confirm/export）；Step5 写 eval 计划（20 条 golden cases：5 个功能完整需求、5 个模糊需求、5 个已知风险、5 个红队样本）；Step6 写复盘模板（目标/取舍/失败/成本/下一步）。"
modules:
  - id: "d12-m1"
    title: "Capstone 范围小但完整"
    concept: "integration"
    idea: "Capstone 候选题要能在 1-2 周内完成端到端闭环，覆盖用户任务、AI 编排、工具或检索、结果呈现、日志、eval 和失败处理，但不追求完整平台。"
    whyItMatters: "范围太大做不完，范围太小证明不了产品能力。\"能演示但没有 eval\"和\"有 eval 但跑不起来\"都不算完整闭环。"
    engineerLens: "使用\"小而完整\"测试：如果能在 demo 中走完一条完整用户路径（从输入到引用输出再到失败处理），且每一步都有 trace，就是合格范围。反之，就继续删功能。"
    productionExample:
      context: "某内容团队把\"智能文案助手\"定为 capstone，最初包含 9 个功能（生成、改写、翻译、多语言、品牌声音、AB 测试、SEO 分析、历史记录、团队协作）。"
      whatTheyDid: "用\"删一半\"原则收敛到 3 个功能（生成草稿、品牌声音引导、SEO 关键词注入），明确非目标（翻译/协作不在 P0），保留完整的 trace、引用、eval 和回滚路径。"
      outcome: "在 8 天内交付了可验收的 demo，eval 覆盖 15 条 golden cases，产品验收通过率 93%。"
    counterExample:
      context: "另一个团队做\"万能 AI 助手\"，输入框 + 回答区，功能列表 20 项。"
      antiPattern: "没有明确用户场景、没有数据契约、没有 eval、没有失败态，9 天内没有一个功能走完完整路径。"
      consequence: "Demo 只能演示最简单的单轮问答，无法证明任何 P0 能力，项目被要求重做。"
    pitfalls:
      - symptom: "功能不断增加，范围边界模糊"
        fix: "在 problem statement 写死 non-goals 并在每次评审时检查边界是否被突破；突破边界的功能进入 backlog 而非当前 capstone"
      - symptom: "每个功能都\"差不多做完了\"但没有一个走完端到端"
        fix: "定义\"完整性标准\"：功能完整 = 输入 + 处理 + 输出 + 失败态 + trace，缺一不算完整"
  - id: "d12-m2"
    title: "交付计划把学习成果变成工程项目"
    concept: "evidence"
    idea: "交付计划包含里程碑、接口契约、数据契约、风险缓解、测试和 eval 前置、部署说明和复盘模板，而不是功能愿望清单。"
    whyItMatters: "没有交付计划，capstone 最终会变成若干孤立 demo；有了计划，每个里程碑都有可运行产物，可以被独立验收。"
    engineerLens: "把里程碑拆成：M1=数据契约与接口 schema（可做 mock 测试）；M2=最小功能路径可跑（端到端 happy path）；M3=eval 与失败处理（golden set 通过率 >= 目标）；M4=部署与复盘文档。每个里程碑有退出条件。"
    productionExample:
      context: "\"内部知识助手\"团队写了交付计划：M1 明确 document schema + retrieval API + citation schema；M2 实现上传、检索、回答、引用展示；M3 用 20 条 golden cases 测试并修复失败；M4 部署到内部 VPN，写复盘文档。"
      whatTheyDid: "每个里程碑有验收 checklist（M1：mock API 通过所有 schema 校验；M2：happy path 截图 + trace id；M3：golden set 通过率 >= 85%；M4：5 个真实用户测试报告）。"
      outcome: "项目按计划 12 天交付，M3 测试发现了 3 个引用权限漏洞，在 M4 前修复，避免了生产事故。"
    counterExample:
      context: "另一个团队写了\"功能计划\"：第 1 周做搜索，第 2 周做回答，第 3 周做 UI。"
      antiPattern: "没有接口契约、没有 eval 计划、没有失败处理、没有验收标准，每周末无法判断完成度。"
      consequence: "第 3 周发现搜索结果和回答格式不兼容，需要重写数据层，实际交期延误 2 周。"
    pitfalls:
      - symptom: "计划写完后没有人遵守"
        fix: "把里程碑验收条件写进 PR 模板，只有 checklist 全部打钩才能 merge 对应 milestone 的 PR"
      - symptom: "风险都在最后才暴露"
        fix: "M1 完成后做一次风险评审，提前识别技术风险（如向量库延迟不达标），并在 M2 做验证性 spike"
  - id: "d12-m3"
    title: "部署方案证明产品能被使用"
    concept: "release"
    idea: "部署方案明确环境变量、密钥管理、数据库、队列、日志、限流、灰度策略和回滚路径。P0 可以本地优先，但仍要声明未来扩展接口。"
    whyItMatters: "\"本地能跑\"是开发者的主观感受，\"能被真实用户使用\"需要可观测性、错误处理、密钥隔离和回滚能力来保证。capstone 的部署方案是工程能力的一部分证据。"
    engineerLens: "最小部署 checklist：① 密钥通过环境变量注入（不进代码）；② 服务有 /health 接口；③ 有结构化日志（request id + trace id + status）；④ 有 rate limit 保护；⑤ 有回滚路径（至少是\"重启上一个 image\"）。"
    productionExample:
      context: "\"客服 RAG\"capstone 团队第一次部署时把 API key 写进了 Docker image，并且没有日志格式约定。"
      whatTheyDid: "重建部署：.env 文件通过 CI secrets 注入，添加 /health 检查，所有请求输出 JSON 日志（含 user_id、query_hash、retrieval_count、latency_ms），用 nginx 做限流（每 IP 10 req/min），编写 rollback.sh（拉取上一个 image tag 并 docker-compose up）。"
      outcome: "生产小测（5 个内部用户）期间发生一次模型超时事件，rollback.sh 在 90 秒内恢复服务，日志清晰定位到冷启动问题。"
    counterExample:
      context: "另一个团队的 capstone 把 OPENAI_API_KEY 写入 next.config.js 并推送到公开仓库。"
      antiPattern: "密钥泄露，没有日志，没有限流，回滚方式是\"手动改代码再 push\"。"
      consequence: "key 在 48 小时内被扫描工具发现，账户被滥用产生 $340 额外费用，需要重置所有密钥并通知用户。"
    pitfalls:
      - symptom: "环境变量在本地和生产不一致导致功能差异"
        fix: "使用 .env.example 文件列出所有必需变量，并在 CI 启动时检查必需变量是否存在"
      - symptom: "日志太多（噪声）或太少（无法 debug）"
        fix: "只记录三类事件：request started（含 trace id）、request completed（含 latency）、error（含 stack trace 摘要）；其余用 debug 级别"
  - id: "d12-m4"
    title: "复盘让一次项目变成长期能力"
    concept: "review"
    idea: "复盘记录目标达成情况、架构取舍原因、失败样本与根因、成本与延迟数据、eval 结果和下一步路线图，而不是情绪总结或成功截图集。"
    whyItMatters: "没有结构化复盘，每次项目积累的经验留在个人脑中，无法传承；结构化复盘把一次实践变成可查询的方法资产，直接影响下一个项目的起点高度。"
    engineerLens: "复盘模板至少包含 6 个字段：① 目标与实际达成对比；② 架构取舍（选了什么、为什么不选另一个）；③ 失败与修复（每个失败的根因一句话）；④ 成本与延迟数据（token 费用、P95 延迟）；⑤ eval 结果（golden set 通过率、失败分类）；⑥ 下一步（最高优先级 3 项，带时间估计）。"
    productionExample:
      context: "\"PRD 评审 Agent\"团队完成 capstone 后进行结构化复盘，发现真实使用中检索质量是主要问题而非生成质量。"
      whatTheyDid: "在复盘文档中记录：目标（节省 PRD 审查时间 50%）vs 实际（节省 38%，差距来自错误引用）；架构取舍（选 pgvector 而非 Pinecone，原因：权限过滤更灵活）；3 个主要失败（引用位置不准确 / 需求模糊时输出质量下降 / 大文档处理超时）；成本数据（每次检索 $0.0023，每次生成 $0.011）；下一步（改进 chunk 策略以提升引用精度）。"
      outcome: "下一个项目团队直接从复盘中提取了 chunk 策略改进方案，节省约 3 天探索时间。"
    counterExample:
      context: "另一个团队的复盘是：\"项目完成了，大家都很努力，下次继续加油。\""
      antiPattern: "没有数据、没有失败根因、没有下一步行动，复盘文档一周后没有人再查阅。"
      consequence: "下一个类似项目重复了相同的 3 个架构错误，多花了约 5 天时间。"
    pitfalls:
      - symptom: "失败样本没有根因，只写\"模型效果不好\""
        fix: "把每个失败归类到 4 个根因之一：数据质量、检索策略、提示设计、系统约束；每个根因写一条可操作的改进动作"
      - symptom: "成本数据没有记录，下一个项目无法估算预算"
        fix: "在每次 eval 运行结束时自动记录 token 消耗和 API 费用到 metrics.jsonl，复盘时直接引用"
