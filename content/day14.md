summary: "为生产 RAG 系统设计可追溯、可权限过滤、可更新、可删除的知识摄取链路与 metadata 契约。"
capabilityGoal: "能写出文档 metadata schema，并说明更新、撤回、权限变更如何沿派生链（解析/chunk/embedding/缓存/引用/trace）传播。"
verifiableOutput: "知识生命周期图（含 7 个节点的 lifecycle 架构）与 metadata contract（含 owner/tenant/acl/version/validity/deletion-state 字段）。"
mentalModel: "知识进入系统后会派生出解析、chunk、embedding、缓存和引用；任何更新、撤回或权限变更都必须沿派生链传播，缺失任何一环都会形成残留风险。"
walkthrough: "企业 HR 撤回旧版差旅报销制度。Step1 系统接收到 delete event（source_id=policy-2022-travel）；Step2 定位所有派生资产：原文（对象存储）、解析结果（PostgreSQL）、2847 个 chunk（pgvector）、36 个 embedding（pgvector）、11 条缓存回答（Redis）、9 条历史 trace 中的引用（MongoDB）；Step3 启动幂等补偿任务（compensation_job_id=uuid，状态可查询，失败重试）；Step4 清理所有派生资产并写入审计记录（保留删除事实，不保留敏感原文）；Step5 在已有 trace 中标记引用状态为 DELETED，后续检索不再召回；Step6 若有用户再次查询同一政策，系统给出\"该政策已撤回，请联系 HR 获取最新版本\"的拒答响应。"
modules:
  - id: "d14-m1"
    title: "数据契约让知识进入系统后可治理"
    concept: "metadata"
    idea: "数据契约规定每份知识文档必须携带的 metadata 字段：source_id、owner、tenant_id、classification、access_scope（ACL）、effective_at、expires_at、version、source_checksum 和 deletion_state。没有契约，知识进入系统后就变成不可追踪的黑盒。"
    whyItMatters: "RAG 失败的根源往往不是检索算法，而是数据生命周期失控：旧制度没有下线、权限没有进入索引、引用没有版本、删除只删了原文件。有了 metadata 契约，知识的每个生命周期事件都有可查询的依据。"
    engineerLens: "把 metadata schema 当作知识系统的第一层 API——就像数据库表的 DDL，入库前必须校验。权限字段（access_scope、tenant_id）必须进入向量索引的 metadata filter，不能只在回答后让模型决定是否显示。"
    productionExample:
      context: "某法律科技公司把客户合同、内部政策和公开法规混入同一向量库，没有 tenant 字段和权限过滤。A 客户查询时检索到了 B 客户的合同片段。"
      whatTheyDid: "引入强制 metadata schema：每份文档在入库时必须通过 schema 校验（缺失字段则拒绝入库），tenant_id 和 access_scope 作为 pgvector 的 WHERE 条件参与检索；version 字段在文档更新时递增，旧版本切换为 superseded 状态。"
      outcome: "跨租户数据泄露事件降至零；文档版本管理使合规审计时间从 3 天降至 4 小时（直接查询 version 历史即可）。"
    counterExample:
      context: "另一个企业知识库只保存了文件名和向量 ID，没有 owner、tenant、版本或权限字段。"
      antiPattern: "知识进入系统后失去所有元信息，无法追踪谁上传了什么、对谁可见、何时生效。"
      consequence: "员工查到了其他部门的薪酬谈判文档（不同部门同放一个 namespace），公司面临内部合规风险。"
    pitfalls:
      - symptom: "知识入库后无法知道某条回答来自哪份文档的哪个版本"
        fix: "在 chunk 级别保存 source_id + version + chunk_index，引用时携带完整 chunk 标识符"
      - symptom: "权限字段在入库后被遗忘，检索时没有过滤"
        fix: "把权限过滤封装成检索函数的必选参数（非可选），无法绕过；在集成测试中验证跨租户隔离"
  - id: "d14-m2"
    title: "新鲜度控制避免旧知识伪装成事实"
    concept: "freshness"
    idea: "向量相似度反映语义接近程度，不反映内容有效性。新鲜度需要通过 effective_at（生效时间）、expires_at（过期时间）、version（版本号）和业务状态（active/superseded/archived/deleted）共同参与检索过滤和重排。"
    whyItMatters: "法规、合同、产品定价、内部政策的\"过期\"方式截然不同：法规按生效日期，合同按签署日期，产品定价按版本，内部政策按 HR 撤回动作。单一的按时间排序不能覆盖这些业务场景；过期知识被模型引用会直接制造生产事故（如错误的报销上限或已废止的条款）。"
    engineerLens: "新鲜度过滤应在检索阶段生效（metadata WHERE 条件），而不是在回答后让模型判断\"这个政策是否还有效\"。检索函数应支持传入 validity_window（查询时刻的有效时间戳）并自动过滤 expires_at < now 或 effective_at > now 的文档。"
    productionExample:
      context: "某零售企业的商品定价 RAG 系统在旧价格表未下线的情况下，持续向销售代表推荐已涨价 15% 的旧价格，造成多次订单纠纷。"
      whatTheyDid: "在 metadata schema 中添加 effective_at 和 expires_at；价格表入库时记录生效日期；检索时自动过滤\"not yet effective or already expired\"；对于版本迭代（新价格表替代旧价格表），旧版本状态改为 superseded，检索时排除 superseded 文档。"
      outcome: "旧价格引用错误率从 6.2% 降至 0.1%（极少数边界情况为时区问题，已添加 UTC 校验）。"
    counterExample:
      context: "另一家公司的知识库里有 2019 年的内部制度文档，没有过期标记；新政策和旧政策共存于同一索引，模型按相似度选择。"
      antiPattern: "检索只看向量分数，旧政策可能比新政策的语义更接近查询，导致旧政策被频繁引用。"
      consequence: "员工按旧报销政策填写单据，财务系统拒绝处理，产生大量返单；HR 不得不发全员邮件澄清，信任度下降。"
    pitfalls:
      - symptom: "不同业务域的文档混用同一个 expires_at 逻辑（如法规按时间，合同按状态）"
        fix: "在 metadata 中增加 freshness_strategy 字段（time-based / status-based / version-based），检索层按策略分别处理"
      - symptom: "文档更新后新版本和旧版本同时存在，检索召回混合版本"
        fix: "文档更新时先把旧版本状态改为 superseded，再插入新版本；使用事务保证原子性"
  - id: "d14-m3"
    title: "删除传播必须覆盖所有派生资产"
    concept: "deletion-propagation"
    idea: "源文件撤回后，系统必须定位并清理（或 tombstone）所有派生资产：原始解析结果（数据库）、所有 chunk（向量库）、embedding（向量库）、检索缓存（Redis/Memcached）、历史 trace 中的引用标记（MongoDB/日志）。只删原文件会让系统继续从派生资产中回答，形成\"幽灵知识\"。"
    whyItMatters: "派生资产清理是法律合规（如 GDPR 的\"被遗忘权\"）和内部信息安全的要求，也是知识库可信度的基础。删除事件必须是幂等的（重复执行结果相同），因为网络失败或系统故障可能导致补偿任务需要重试。"
    engineerLens: "为每个 source_id 维护一张派生资产清单（derived_asset_manifest）：记录 document_id、chunk_ids[]、embedding_ids[]、cache_keys[]、trace_ids[]。删除事件触发补偿任务时，按清单逐项清理；任务状态（pending/running/completed/failed）可查询；失败时自动重试（指数退避，最多 5 次）。"
    productionExample:
      context: "某医疗机构的知识库在患者撤回知情同意后，只删除了对象存储中的原始 PDF，但向量库中的 chunk 和 Redis 缓存中的检索结果仍然存在。"
      whatTheyDid: "引入 derived_asset_manifest 表（PostgreSQL）记录每个 source 的所有派生资产 ID；删除事件触发异步补偿任务（Celery），按 manifest 逐步清理（chunk、embedding、cache、trace 标记）；每步清理结果写入审计日志（保留删除事实，不保留原始内容）。"
      outcome: "合规审计时能够证明删除覆盖了所有派生资产（通过 audit log 溯源）；GDPR 删除请求平均处理时间从 72 小时降至 4 小时。"
    counterExample:
      context: "另一家公司的删除流程是手动的：IT 接到请求后删除对象存储文件，向量库从未被清理过。"
      antiPattern: "向量库中积累了 3 年的\"已删除\"文档的 chunk，在相似查询时仍被召回。"
      consequence: "外部审计发现系统仍能回答基于\"已删除\"内容的问题，公司面临 GDPR 违规风险，罚款风险。"
    pitfalls:
      - symptom: "删除补偿任务失败后没有重试，形成部分删除的不一致状态"
        fix: "使用幂等补偿任务（每个清理步骤检查是否已执行），配合指数退避重试；任务失败后触发告警，人工复核"
      - symptom: "审计日志保存了被删除内容的原始文本（违反隐私要求）"
        fix: "审计记录只保存 source_id、chunk_count、deletion_timestamp、operator_id，不保存内容摘要或原始文本"
  - id: "d14-m4"
    title: "引用要指向可复核的具体证据"
    concept: "permission-filter"
    idea: "知识系统的引用不是一个 URL，而是一个包含 document_id、document_version、chunk_id、位置锚点（start_char/end_char 或 page_number）、权限状态（用户是否有权查看原文）和生成时快照摘要的结构化对象。"
    whyItMatters: "监管机构、审计员或用户追问\"这个答案的依据是什么\"时，必须能定位到具体的文档版本和段落，而不是一个可能已失效的链接。引用也是 grounding 校验的前提——没有 chunk_id，就无法检查答案是否真的来自引用段落。"
    engineerLens: "引用 schema 设计：{ source_id, document_version, chunk_id, location: { start_char, end_char }, access_permitted: bool, snapshot_at: timestamp, content_hash: string }。前端展示引用时检查 access_permitted；如果为 false，显示\"您无权查看原文，请联系管理员\"。"
    productionExample:
      context: "某法律助手产品最初只在回答中返回文档标题，用户质疑某条法律建议的依据时无法验证。"
      whatTheyDid: "在 answer API 的响应中添加 citations[] 数组，每条引用包含 source_id + version + chunk_id + 原文摘录（50 字）+ 位置锚点；前端引用组件支持展开查看完整段落并跳转到原文位置；对于无权限的引用显示\"来源受限，已记录请求\"。"
      outcome: "律师用户的信任评分从 3.2/5 提升至 4.4/5（主要驱动因素：可以验证每条建议的原文依据）；内部合规审查时间缩短 60%。"
    counterExample:
      context: "另一个知识助手只返回\"参考来源：内部政策库\"，没有具体文档、版本或段落信息。"
      antiPattern: "引用是装饰性的，无法实际复核；如果原文档已更新或撤回，引用的来源和当前内容不一致。"
      consequence: "员工按\"内部政策\"建议操作但被财务拒绝，调查发现引用的是已撤回的旧政策，但系统无法证明是哪个版本，纠纷无法快速解决。"
    pitfalls:
      - symptom: "引用显示的原文摘录与实际生成答案使用的内容不一致"
        fix: "在生成时同步保存 cited_chunk_ids，回答完成后从向量库反查对应 chunk 内容，确保引用与生成上下文一致"
      - symptom: "有权限的用户看到了\"来源受限\"提示，影响使用体验"
        fix: "在检索前预先检查用户的 access_scope 与 chunk 的 access_scope 交集，只返回用户有权访问的 chunk；access_permitted 字段应在检索层而非展示层计算"
