summary: "把\"问模型\"升级成 prompt 合约、检索、引用、评估和反馈闭环，让回答有据可查、有边界可守。"
capabilityGoal: "能为知识型 AI 功能设计完整的 prompt 合约、文档入库规范、混合检索策略和引用校验机制，并用 golden set 评估检索质量。"

mentalModel: "检索提供事实，Prompt 定义规则，模型组织语言；三者缺一不可，任何一环失败都会导致不可信答案。"
walkthrough: "某企业内部合规问答系统需要回答\"这份合同适用哪个版本的数据隐私政策\"。系统先解析合同文件，按章节切块并附加 metadata（地区、政策版本、生效日期、权限标签）；用户提问后，查询改写生成多个检索词，hybrid 检索召回候选，rerank 压缩至最相关的 3 个片段；prompt 合约要求模型只能基于给定片段作答并附带 source_id；生成后的 validator 校验引用覆盖率，不足时触发\"证据不足\"拒答而非强行生成，最终答案带引用标注展示给用户。"
modules:
  - id: "d02-m1"
    title: "Prompt 是可测试的接口合约"
    concept: "prompt-contract"
    idea: "生产 prompt 不是聊天开场白，而是定义任务、输入格式、输出 schema、约束规则、拒答条件和验收示例的工程契约，可版本化、可评估、可回归。"
    whyItMatters: "没有合约的 prompt 让模型输出不可预期；每次改动都靠感觉，无法量化改善，也无法在版本之间做回归对比。"
    engineerLens: "像设计 REST API 一样设计 prompt：system message 定义角色和硬约束，developer 指令定义任务流程和输出 JSON schema，few-shot 示例固定难以描述的边界判断，并记录版本号和对应的验收样例。"
    productionExample:
      context: "某 HR 科技公司的薪酬政策问答系统，面向 3000 名员工，覆盖 12 个地区的薪酬规则。"
      whatTheyDid: "为问答 prompt 设计合约：输入字段（用户问题、检索到的政策片段列表和用户地区）、输出 schema（answer、citations 数组、confidence 枚举、refused_reason 可选字段）、拒答规则（证据片段不覆盖问题时必须设置 refused_reason）、版本号和 10 条 golden QA 对。"
      outcome: "Prompt 合约化后，工程师可以在不改逻辑的情况下独立更新 schema 和拒答规则；六次迭代中每次都有回归数据对比，质量问题在上线前全部发现。"
    counterExample:
      context: "某创业公司的客服机器人，prompt 是一段手写的自然语言描述，无版本控制。"
      antiPattern: "每次客服投诉都临时修改 prompt 措辞，没有 schema 和拒答规则，改动后只看一两个 demo 确认\"感觉更好\"。"
      consequence: "半年后 prompt 堆积到 2000 字，不同地区的回答互相矛盾，任何一次改动都可能破坏另一个场景，最终不得不从头重写。"
    pitfalls:
      - symptom: "prompt 中没有输出 schema，模型自由发挥格式"
        fix: "定义 JSON schema，包含所有字段名、类型、是否必填和空值语义"
      - symptom: "prompt 改动没有版本号，无法知道哪次改动造成质量退步"
        fix: "把 prompt 当代码管理：git 版本、changelog 和对应的 golden set 回归结果"
  - id: "d02-m2"
    title: "RAG 是知识注入策略，不是产品形态"
    concept: "chunking"
    idea: "RAG 通过检索把外部知识在本轮上下文中按需注入，适合动态知识、长文档和权限范围不能靠模型参数记忆解决的场景；chunk 质量决定召回上限。"
    whyItMatters: "把整份文档塞进 prompt 既贵又会引入过期信息；chunk 粒度错误会把跨段的关联证据切断，导致召回有词无义。"
    engineerLens: "入库流水线核心决策：按语义边界切分（段落、标题、表格行），每个 chunk 附加 metadata（source_id、版本、地区、生效日期、权限标签），向量和关键词双路索引；入库后用 5~10 条问题验证召回覆盖率再投产。"
    productionExample:
      context: "某保险公司的理赔问答系统，知识库包含 800 份产品条款，每份条款有多个版本，每个版本按地区和险种分类。"
      whatTheyDid: "按条款章节切块（目标 chunk 大小 400~600 token，跨章节加 overlap 128 token），metadata 记录 product_code、region、effective_date 和 permission_level；向量索引用于语义检索，BM25 索引用于产品编号和术语精确匹配；上线前用 50 条高频问题验证覆盖率。"
      outcome: "混合索引下首轮召回覆盖率达 91%，比纯向量索引高出 14 个百分点；metadata 过滤避免了跨版本混答问题。"
    counterExample:
      context: "某律所的法规问答原型。"
      antiPattern: "把整份法规 PDF 直接转文本拼进 prompt，没有切块和 metadata，让模型\"自己找相关内容\"。"
      consequence: "输入 token 超限导致截断，新旧法规混在一起，模型无法区分生效日期；项目演示后直接被否决，重新设计耗时 6 周。"
    pitfalls:
      - symptom: "chunk 过大导致单个片段包含多个不相关主题，向量相似度信噪比低"
        fix: "以语义完整性为目标切分，段落级优先，表格和列表作为独立 chunk，并测量 chunk 内的主题纯度"
      - symptom: "没有 metadata，无法按版本或权限过滤"
        fix: "入库时强制要求 source_id、版本号、生效日期和权限标签，检索时前置 metadata 过滤"
  - id: "d02-m3"
    title: "检索质量要用 golden set 调参"
    concept: "retrieval"
    idea: "检索评估要分离召回质量（是否找到正确证据）和答案质量（模型是否用了证据），两者需独立测量；没有 golden set，就不知道该改 chunk、embedding、rerank 还是 prompt。"
    whyItMatters: "检索错了，模型会在错误证据上生成看似合理的答案，而这种\"幻觉\"比明显错误更难发现，危害更大。"
    engineerLens: "先建 20 条高频问题的 golden set（每条包含问题、期望证据的 chunk_id 列表、可接受答案范围和失败备注）；按 recall@k、MRR 和 NDCG 评估检索，按 citation_coverage 和 factual_accuracy 评估生成；调参顺序：先 chunk，再 embedding，再 rerank，最后 prompt。"
    productionExample:
      context: "某 SaaS 公司的技术支持知识库，月均处理 15000 条技术问题。"
      whatTheyDid: "建立 200 条 golden set，覆盖错误码查询、配置说明和版本迁移三类场景；top-5 召回率从 58% 优化到 87%，主要改进来自把代码块和文档正文分开切块、加入产品版本过滤和引入 rerank 模型；整个调参过程耗时 3 周，有数据支撑每一步改动。"
      outcome: "技术问题首次自动解决率从 41% 提升至 69%，工单转人工量减少 35%。"
    counterExample:
      context: "某团队在没有 golden set 的情况下调整 chunk 大小和 embedding 模型，每次改动都靠 demo 判断。"
      antiPattern: "随机选 3 个问题测试，效果看起来好就发布；没有跨场景覆盖，不追踪召回率变化。"
      consequence: "一次 embedding 模型升级导致特定产品名称查询召回率下降 40%，上线后才被发现，回滚耗费 2 天。"
    pitfalls:
      - symptom: "golden set 只有 3~5 条，无法覆盖边界场景"
        fix: "从真实用户问题中抽取高频和高失败率问题，构建至少 20 条；每月补充新增失败案例"
      - symptom: "只看最终答案质量，忽略召回质量"
        fix: "分层测量：先评估检索（chunk 是否命中 golden evidence），再评估生成（引用是否覆盖核心结论）"
  - id: "d02-m4"
    title: "Grounding 让答案可追溯"
    concept: "grounding"
    idea: "生产 RAG 的回答必须区分三种来源：有引用支持的事实、模型基于证据的推断、以及模型参数中的知识；不支持的结论必须拒答或明确标注不确定性。"
    whyItMatters: "用户真正需要的是可信、可追溯的答案，而不是流畅的文本；没有引用的 RAG 答案与没有 RAG 的答案在信任度上没有本质区别。"
    engineerLens: "在生成后的 validator 中检查三项：citation_coverage（核心结论是否有 source_id 支撑）、schema_valid（输出格式是否符合定义）和 unsupported_claim（是否有无证据结论）；不依靠 prompt 祈祷，而是系统侧强制校验。"
    productionExample:
      context: "某医疗健康平台的药品信息问答系统，日均处理 8000 条用户咨询。"
      whatTheyDid: "每条答案强制要求 citations 字段列出支撑段落的 chunk_id；后处理 validator 用规则检查答案中的关键信息点（剂量、禁忌）是否都有对应引用；validator 失败则答案进入\"专业人员确认\"队列而非直接返回；每周人工复核 100 条 validator 通过的答案。"
      outcome: "无引用支撑的关键信息点被拦截率达到 96%，用户对答案的信任度评分（可信 vs 不可信）从 61% 提升至 83%。"
    counterExample:
      context: "某智能客服系统的 FAQ 问答，用 RAG 召回但没有引用校验。"
      antiPattern: "模型把召回的多个 FAQ 内容混合重写，答案流畅但无法追溯具体来源，且会把不同版本的政策混合输出。"
      consequence: "用户引用 AI 答案要求特殊处理，但客服无法确认答案来源，导致多起客诉升级；产品团队被迫在所有 AI 回答上加免责声明，用户信任度大幅下降。"
    pitfalls:
      - symptom: "要求模型\"提供引用\"但不校验引用是否真实存在"
        fix: "validator 必须验证 source_id 在本轮召回集中确实存在，防止模型捏造 citation"
      - symptom: "模型无法找到证据时仍强行给出答案"
        fix: "在 prompt 合约中明确\"证据不足时输出 refused_reason 字段并停止生成\"，并在 validator 中强制检查"
decisionLayers:
  - id: "dl02-1"
    name: "入库链路"
    question: "知识如何转化为可信赖的可检索资产？"
    choices:
      - name: "语义切分加 metadata"
        description: "按语义完整性切块，附加 source、版本、权限和时间 metadata；是检索精度的基础，不能靠后处理弥补。"
        example: "政策文档按章节切块，每块标注地区、生效日期和权限等级。"
      - name: "混合索引"
        description: "向量索引处理语义相似查询，BM25 索引处理精确术语、产品编号和错误码；单一索引在生产环境中通常不够。"
        example: "技术文档用向量处理\"怎么配置\"类问题，用 BM25 处理\"ERR_4023 是什么\"类问题。"
  - id: "dl02-2"
    name: "查询链路"
    question: "问题如何找到正确证据？"
    choices:
      - name: "查询改写加混合检索"
        description: "用户原始问题不总是最优检索词；改写为多个查询变体后分别检索再合并，提升召回覆盖率。"
        example: "\"退款需要多久\"改写为\"退款周期\"\"退款时效\"\"refund processing time\"三条查询。"
      - name: "Rerank 压缩证据"
        description: "top-K 召回后用 cross-encoder rerank 从大候选集压缩到 top-N，提升精度并减少 prompt 噪声。"
        example: "top-20 召回后 rerank 保留最相关的 top-5，减少 context 窗口占用。"
  - id: "dl02-3"
    name: "答案链路"
    question: "如何让答案可信可追溯？"
    choices:
      - name: "强制引用加 validator"
        description: "输出 schema 要求所有核心结论附带 source_id；生成后 validator 校验引用覆盖率，不达标则触发拒答。"
        example: "回答中每个关键事实点都带 [src:chunk_042] 标注，validator 检查覆盖率超过 80%。"
      - name: "拒答兜底"
        description: "证据不足时不能强行生成；明确拒答并说明缺口，必要时转人工，比\"可能是这样\"更可信。"
        example: "找不到覆盖用户问题的政策版本时，返回\"当前知识库缺少该版本政策，请联系合规部门\"。"
