modules:
  - id: "d19-m1"
    title: "AI 资产必须版本化才能追溯行为变化"
    concept: "versioning"
    idea: "Prompt 模板、工具 schema、检索配置、eval 数据集、模型路由和策略规则都必须有版本号，且版本号必须出现在每次请求的 trace 中，才能从线上问题追溯到具体资产。"
    whyItMatters: "没有版本号，团队无法回答\"昨天还对今天错了，中间改了什么\"这个最基本的调试问题；版本化是所有回滚、审计和对比实验的基础。"
    engineerLens: "建立 release manifest（JSON 文件，存入 Git）：每次发布时记录 release_id、prompt_template_version、model_route_version、retrieval_index_version、tool_schema_version、eval_dataset_version 和 owner；每次请求的 trace 必须携带当前 release_id，使线上问题可以精确追溯到对应发布版本。"
    productionExample:
      context: "某企业知识问答平台连续 3 次出现\"回答行为突变\"问题，但团队无法定位是 prompt、模型路由还是数据集变更导致。"
      whatTheyDid: "引入 release manifest，将所有 AI 资产版本统一记录；每次 trace 携带 release_id；出现问题时只需查询 trace 中的 release_id，再对比两次 manifest 的差异即可定位变更点；同时对 prompt 文件引入 Git 分支保护，禁止直接 push 到 main，必须通过 PR 审批。"
      outcome: "下次行为异常事故中，30 分钟内通过 release_id diff 定位到 prompt 版本变更；而此前同类问题平均需要 4 小时排查。"
    counterExample:
      context: "某 SaaS 平台运营同学直接在管理后台编辑系统 Prompt，修改立即生效，无版本记录。"
      antiPattern: "Prompt 变更不经评审，无版本号，无 trace 关联，无回滚机制。"
      consequence: "某次 Prompt 修改引入了未预期的行为变化（在某类问题上开始拒绝回答），用户投诉后团队无法复现原始行为；花费 2 天尝试重写旧 Prompt 效果仍有差异。"
    pitfalls:
      - symptom: "版本号只是数字递增，无法知道两个版本间的差异"
        fix: "使用 Git commit hash 作为版本号，或在 manifest 中记录 diff_summary，说明本次变更内容"
      - symptom: "eval 数据集版本未纳管，测试结果依赖于哪个版本的数据集不清晰"
        fix: "在 CI 测试命令中明确指定 eval_dataset_version，测试结果和数据集版本一起记录"
      - symptom: "版本号只在文档中记录，trace 中没有携带"
        fix: "在应用启动时从环境变量或配置服务读取 release_id，注入到所有 trace span"
  - id: "d19-m2"
    title: "发布门禁连接质量安全和成本"
    concept: "release-gate"
    idea: "发布前必须通过机器可验证的检查项（eval 达标、红队通过、成本预算、延迟 SLO、权限回归和回滚路径确认），演示成功不能替代门禁。"
    whyItMatters: "没有门禁的 AI 资产发布等于在生产环境做实验；AI 变更的影响范围广（影响所有查询），一次错误发布可能在几分钟内影响数千用户。"
    engineerLens: "发布门禁 checklist（可机器验证的条目优先）：① eval 命中率不低于上版本基线（CI 自动运行并阻断）；② 红队样本（至少 10 个对抗 case）无新增失败；③ 成本预估（输入/输出 token 预算）不超过上版本 20%；④ p95 延迟预估在 SLO 内；⑤ 权限边界回归（高权限 prompt 无法被低权限用户触发）；⑥ 回滚路径已验证（确认上一版本资产可用且可一键激活）；门禁失败时阻断发布并通知 owner，不允许绕过。"
    productionExample:
      context: "某 AI 代码审查工具在更新 prompt 时未经 eval 测试直接发布，新版 prompt 对 Python 代码有系统性误判。"
      whatTheyDid: "引入 CI 集成的 eval gate（基于 LangChain Evals 框架）：每次 prompt PR 合并时自动运行 500 个回归样本，要求准确率不低于当前主干 -2%；红队 checklist 要求安全工程师在 PR 中签署\"已检查注入向量\"；成本预算超 20% 时需要产品总监审批。"
      outcome: "在引入 eval gate 后的 6 个月内，两次 prompt 变更被门禁自动阻断（其中一次发现了严重的漏分类问题），零错误发布到生产。"
    counterExample:
      context: "某团队以\"发布节奏加快\"为由跳过 eval gate，采用\"上线后观察\"策略。"
      antiPattern: "发布门禁可由任意工程师绕过（--skip-gate 参数），观察窗口为 24 小时；高风险变更和低风险变更没有区分对待。"
      consequence: "一次改动在高并发场景下触发了上下文长度超限，造成全平台 30% 请求失败；观察窗口结束时才发现问题，影响持续近 2 小时。"
    pitfalls:
      - symptom: "门禁检查项全部依赖人工填写，无法自动化验证"
        fix: "将 eval 结果、成本预估和延迟测试全部输出为机器可读格式（JSON），CI 自动读取并判断是否达标"
      - symptom: "所有变更走同一套门禁，紧急安全补丁也需要等待完整 eval"
        fix: "根据风险等级设置不同门禁路径：安全补丁可简化 eval 但必须安全审批；高风险 prompt 变更需要双人评审"
      - symptom: "门禁通过后没有记录证据，事后无法证明当时已检查"
        fix: "每次发布生成发布证明文件（包含 eval 结果、红队签署和审批记录），存入 Git 的 releases 目录"
  - id: "d19-m3"
    title: "灰度发布控制 AI 变更的风险暴露"
    concept: "gray-rollout"
    idea: "AI 资产变更应对低风险流量（内部用户/小比例任务）逐步放量，使用 trace 指标决定是否继续扩量或回滚；不是所有变更都适合灰度，安全漏洞应直接阻断。"
    whyItMatters: "离线 eval 覆盖的是已知分布，线上真实流量的分布和边界情况往往超出 golden set；灰度是连接离线评估和全量发布之间的风险缓冲层。"
    engineerLens: "灰度策略设计：根据 user_tier 或 request_hash 将流量分桶（5% → 20% → 50% → 100%）；每阶段观察窗口至少 48 小时；观察指标：质量 SLI、成本 delta、p95 延迟 delta 和投诉率；异常阈值（如 SLI 下降 >5%）时自动回滚；灰度期间每条请求 trace 标记当前实验分组 ID，方便对比分析。"
    productionExample:
      context: "某 RAG 平台将检索重排策略从 BM25 升级为语义重排，离线 eval 显示命中率提升 8%，但对线上真实分布不确定。"
      whatTheyDid: "使用 feature flag 服务（LaunchDarkly）按 user_id hash 灰度；阶段：内部用户 100%（第 1-3 天）→ 5% 外部用户（第 4-10 天）→ 20%（第 11-17 天）→ 100%（第 18 天）；每天自动生成灰度对比报告（命中率、成本、p95 延迟和投诉率）；第 6 天发现某类长尾查询退化 4%，暂停扩量，修复后继续。"
      outcome: "全量后命中率提升 7.2%（接近离线 8%），无意外成本超支，用户投诉率无变化；灰度阶段捕获并修复了 1 个回归问题。"
    counterExample:
      context: "某平台将安全策略 prompt 的权限越权漏洞修复推入灰度，计划观察 7 天。"
      antiPattern: "将安全漏洞修复也纳入常规灰度流程，7 天内 5% 流量仍使用有漏洞的 prompt。"
      consequence: "安全研究员在灰度窗口期发现了该漏洞并公开披露；虽然影响范围是 5%，但公关影响是 100%；安全漏洞修复应立即全量而不是灰度。"
    pitfalls:
      - symptom: "灰度分组不包含 trace 标签，无法区分灰度组和对照组的指标"
        fix: "灰度时在每个请求 trace 中注入 experiment_id 和 variant（treatment/control），监控按 variant 分组"
      - symptom: "灰度观察窗口太短，没有覆盖流量低谷（如周末）"
        fix: "至少覆盖完整的 7 天自然周期，包含高峰和低谷期"
      - symptom: "灰度结束后直接全量，未保留快速回滚能力"
        fix: "全量后继续保持 feature flag 有效，保留至少 2 周的快速回滚能力"
  - id: "d19-m4"
    title: "责任边界决定谁能发布和回滚"
    concept: "ownership"
    idea: "每类 AI 资产（prompt、工具、数据集、检索配置）必须有明确的 owner、审批人、灰度观察人和回滚责任人；没有明确 owner 等于没有人对质量和事故响应负责。"
    whyItMatters: "AI 资产跨越产品、研发、安全和运营多个团队；模糊的所有权会导致\"大家都可以改，出事无人负责\"的状态，在事故响应时表现为没有人知道谁可以回滚。"
    engineerLens: "在 release manifest 中为每类资产记录 owner（日常维护）、approver（评审和发布批准）、security_reviewer（安全相关变更审批）和 rollback_owner（事故时有权回滚）；在 runbook 中明确 on-call 联系方式；定义哪些资产可由产品更新（低风险内容 prompt）、哪些需研发审批（工具 schema）、哪些需安全审批（权限策略、用户数据访问）。"
    productionExample:
      context: "某 AI 内容审核平台有 3 个团队共同维护 prompt、规则库和模型路由，出事故时没有人清楚谁负责回滚。"
      whatTheyDid: "为每类资产建立 OWNERS 文件（参考 Google Monorepo 模式）：prompt 目录 owner 是内容团队，工具 schema owner 是 API 团队，权限策略 owner 是安全团队；每次 PR 必须由对应 owner 审批；runbook 中明确\"回滚请联系 rollback_owner，电话：xxx\"。"
      outcome: "下次事故时，on-call 工程师 5 分钟内联系到 rollback_owner 并授权回滚；整个过程有审批记录，符合合规要求。"
    counterExample:
      context: "某小团队认为大家都了解系统，不设置明确 owner，任何人都可以修改任何资产。"
      antiPattern: "无 OWNERS 文件，无审批流程，无回滚权限记录；\"大家都懂\"是隐性知识，无法在事故时快速调动正确的人。"
      consequence: "核心工程师休假期间出现事故，现场人员无法判断谁有资格回滚，讨论 20 分钟后才开始操作；事后审计发现无授权记录，引发合规风险。"
    pitfalls:
      - symptom: "owner 被标记为整个团队而非具体个人，事故时无法快速联系"
        fix: "owner 必须是具体个人姓名和联系方式，设置备选人（primary + backup）"
      - symptom: "rollback_owner 和 owner 是同一人，该人休假时两个角色都空缺"
        fix: "rollback_owner 和 owner 应是不同人，并设置 backup；On-call rotation 覆盖所有时区"
      - symptom: "审批记录只在聊天工具里，无法在事后审计时提取"
        fix: "审批必须在代码仓库的 PR 或正式工作流系统中完成，保留可检索的历史记录"
