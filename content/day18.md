title: "AI SRE 与事故响应"
summary: "让 AI 功能具备可观测、可降级、可回滚和可复盘的运行能力，通过 SLO、trace、熔断和 runbook 把事故从被动响应转为主动控制。"
capabilityGoal: "能为 AI 功能定义 SLO 和可观测指标，设计端到端 trace，建立熔断回滚机制，并输出可执行的事故 runbook 和无责复盘流程。"
verifiableOutput: "两份 runbook（召回质量骤降和工具超时），以及一次失败 trace 的完整分析报告。"
mentalModel: "AI 事故不是单点报错，是检索、模型、工具、策略和发布变更的交叉失效；trace 是唯一能定位交叉失效的工具。"
walkthrough: "某知识问答平台某天收到大量用户投诉\"答案不对\"。值班工程师查看监控只有 HTTP 200，没有异常。通过端到端 trace 发现检索 span 的召回文档数从 8 降至 1，进一步追溯到当天凌晨的 metadata 过滤规则发布：新规则误将\"internal\"标记文档全部排除，而主要知识库恰好都是该标记。团队执行 runbook：立即关闭新版检索器（kill switch），回滚检索配置到 v3.1.2，在 3 分钟内恢复召回质量；事后无责复盘发现问题根因是发布门禁未检查 golden set 召回数量，将该检查加入发布 CI，并把 10 个失败样本加入回归集。"
modules:
  - id: "d18-m1"
    title: "SLO 用用户可感知结果约束系统"
    concept: "slo"
    idea: "SLO 必须用用户可感知的结果定义（引用准确率、任务完成率、可用延迟），而不是技术层面的 HTTP 200 率；SLI 要可计算，误差预算要能指导节奏决策。"
    whyItMatters: "纯技术指标（CPU、HTTP 200）可以完全正常而用户体验极差；AI 质量故障往往在技术层面无报错，但用户已经收到了错误答案。"
    engineerLens: "为知识问答定义三类 SLI：答案引用命中率（引用锚点指向真实文档片段）、任务完成率（用户未在 10 秒内重新提问）、安全失败率（敏感回答触发人工审核比例）；SLO = 90 天窗口内各 SLI 达标率；错误预算 = SLO 设定值与实际值的差额，用于决定是否可以发布新版本。"
    productionExample:
      context: "某医疗信息平台 AI 问答系统在技术层面无报错，但用户大量投诉\"回答不准确\"。"
      whatTheyDid: "引入\"用户未二次追问率\"作为 SLI（二次追问视为上次回答不满足需求）；设置 SLO 为每周不低于 82%；当 SLI 连续 3 小时低于 75% 时触发页面降级提示（\"当前为摘要模式，建议咨询医生\"），并创建 P1 事故。"
      outcome: "发现了一批因检索版本错误导致的低质量回答；在 SLO 框架下，3 次小事故被识别和修复，避免了一次大规模投诉。"
    counterExample:
      context: "某客服机器人仅监控 HTTP 成功率和平均响应时间，不设 AI 质量 SLO。"
      antiPattern: "技术指标全绿但用户不断投诉答案鸡同鸭讲；团队认为\"监控没问题\"而不立项修复。"
      consequence: "三个月后用户流失率提升 20%，核心投诉量累积成公关危机，才被迫重构可观测体系。"
    pitfalls:
      - symptom: "SLO 目标设置为 99.99% 不切实际，错误预算极小，团队陷入永远无法发布的状态"
        fix: "SLO 目标参考历史数据，新产品初期设宽松目标（如 85%），随成熟度迭代收紧"
      - symptom: "SLI 无法自动计算，只能人工抽查"
        fix: "SLI 必须有对应的自动化计算方式（日志事件、评分模型或用户行为埋点），否则不算有效 SLI"
      - symptom: "SLO 有时间窗口，但超窗口的错误预算不进行回顾"
        fix: "每个 SLO 时间窗口结束时进行回顾，评估是否需要调整目标或加速修复"
  - id: "d18-m2"
    title: "Trace 连接一次请求的输入决策和结果"
    concept: "trace"
    idea: "一次 AI 请求的 trace 必须覆盖前端、网关、检索、模型、工具和策略所有 span，并关联 prompt 版本、模型版本、检索索引版本和工具 schema 版本，才能定位交叉失效。"
    whyItMatters: "聚合指标告知有问题，trace 才能定位问题在哪一跳；AI 系统的特殊性在于同一个 HTTP 200 背后可能对应完全不同质量的回答，只有 trace 能连接行为和输出。"
    engineerLens: "使用 OpenTelemetry，为每次请求生成 trace_id（传递给所有下游）；在每个 span 中记录：prompt_template_version、model_id、retrieval_index_version、tool_schema_version、input_tokens、output_tokens、latency_ms；检索 span 记录 retrieved_count 和 top1_score；工具 span 记录参数和返回 schema；生产环境对 prompt 内容按隐私级别脱敏，保留模板结构。"
    productionExample:
      context: "某 RAG 问答平台偶发\"答案引用了已删除文档\"的问题，用平均指标无法复现。"
      whatTheyDid: "引入 span 级 retrieval_doc_ids 记录，每次检索将实际召回的文档 ID 记入 span；配合删除日志，在 trace 中查找\"检索到已标记为 deleted 的文档 ID\"的 span；发现根因是索引更新有 2 小时延迟窗口，在延迟窗口内删除文档仍可被检索。"
      outcome: "索引更新延迟问题被精准定位，引入 soft-delete 标记实现即时过滤；此前平均 3 天才能复现的问题缩短为 30 分钟内定位。"
    counterExample:
      context: "某 AI 写作助手只记录\"请求成功\"和\"请求失败\"两个事件，无任何中间 span。"
      antiPattern: "日志只有 request_id、status_code 和 total_latency；无法判断延迟来自检索、模型还是工具。"
      consequence: "一次\"生成质量骤降\"事故中，团队花费 6 小时手动对比模型版本、检索配置和 prompt，最终靠猜测回滚，无法证明根因。"
    pitfalls:
      - symptom: "Trace 中记录了原始 prompt 内容，包含用户隐私信息"
        fix: "对 prompt 内容脱敏（替换 PII 为占位符），只保留模板版本和 token 数量；输出内容同理"
      - symptom: "所有请求都 100% 采样，导致 trace 存储成本过高"
        fix: "正常请求用 1-5% 采样；错误请求、慢请求（超过 p95 阈值）和用户投诉请求 100% 采样"
      - symptom: "Trace 只覆盖模型调用，遗漏了检索和工具 span"
        fix: "强制要求所有 I/O 操作（检索、工具调用、外部 API）都必须有对应 span，否则 trace 不完整"
  - id: "d18-m3"
    title: "熔断与回滚优先减少用户影响"
    concept: "circuit-breaker"
    idea: "AI 故障响应的第一优先级是止血（减少受影响用户数），第二优先级是保留证据（供复盘使用），然后才是根因分析；回滚目标必须包括 prompt、模型、数据集、检索配置和工具 schema。"
    whyItMatters: "AI 故障往往会持续生成错误结果（每次调用都返回错误答案），必须先缩小影响面；继续放量排查会使更多用户受到影响，错误样本也会污染反馈循环。"
    engineerLens: "为每个高风险组件设计 kill switch（功能开关，支持运行时关闭，不需要重新部署）；熔断器配置：检索失败率 >20% 时降级为纯 LLM 回答，工具失败率 >30% 时关闭工具调用；每个资产（prompt、索引、模型路由）维护\"安全版本\"记录，支持一键回滚；回滚后 5 分钟内检查关键 SLI 是否恢复，若无恢复继续下一级降级。"
    productionExample:
      context: "某电商 AI 助手在大促期间因商品数据库连接超时导致工具调用失败率飙升至 60%，所有涉及商品查询的回答都显示\"抱歉，无法获取信息\"。"
      whatTheyDid: "工具熔断器（基于 Resilience4j）在失败率超 40% 时自动切换为\"只使用缓存商品摘要回答，提示数据可能非最新\"；同时关闭高风险工具（实时库存查询），保留低风险工具（商品分类浏览）；kill switch 响应 30 秒，比代码回滚快 10 分钟。"
      outcome: "在 90 秒内将受影响用户比例从 100% 降至 18%（仅实时库存查询功能受损）；大促 GMV 影响降至可接受范围；事故全程记录在 trace 中供复盘使用。"
    counterExample:
      context: "某 B2B AI 合规检查平台在模型版本升级后大量输出误判结果，团队继续放量复查问题。"
      antiPattern: "没有 kill switch，没有版本回滚机制；工程师在生产环境实时调试了 2 小时。"
      consequence: "2 小时内 3000 家企业收到了错误合规报告；部分用户已经基于错误报告做出决策；信任损失和法律赔偿金额远超技术修复成本。"
    pitfalls:
      - symptom: "回滚只回滚代码，不回滚 prompt 版本和检索配置"
        fix: "建立 release manifest，包含 prompt_version、model_route_version、index_version 和 tool_schema_version；回滚时必须同时回滚所有关联资产"
      - symptom: "熔断阈值是固定值，正常业务高峰也会误触发熔断"
        fix: "熔断阈值参考历史基线动态调整（如基线失败率 +3σ），或设置绝对量阈值避免低流量时百分比过于敏感"
      - symptom: "降级后没有通知用户，用户误以为功能正常"
        fix: "降级时前端展示降级状态提示（如\"当前使用简化模式\"），并说明限制"
  - id: "d18-m4"
    title: "事故复盘把失败转为系统改进"
    concept: "rollback"
    idea: "有效的 AI 事故复盘必须生成事实时间线、根因分析、行动项（含 owner 和 deadline）和回归样本；复盘结论必须进入发布门禁和测试用例，才能防止同类事故重演。"
    whyItMatters: "没有无责复盘，团队会隐瞒事故细节；没有行动项跟踪，复盘成为形式；没有回归样本，同类问题无法被回归测试覆盖。"
    engineerLens: "事故复盘模板：① 事故摘要（影响范围、持续时间、业务影响）；② 时间线事实（检测、响应、缓解、恢复的精确时间）；③ 根因分析（用 5-why 追溯根因）；④ 行动项（描述、owner、deadline、完成标准）；⑤ 失败样本（从 trace 提取 3-5 个典型失败 case，脱敏后加入回归集）；⑥ 检测与响应改进（SLO 是否需要调整，监控是否需要补充）。"
    productionExample:
      context: "某 AI 法律助手连续两个月出现\"引用了过期法规\"问题，每次复盘结论都是\"加强测试\"，但没有具体行动。"
      whatTheyDid: "引入结构化复盘模板，要求每个行动项必须有机器可验证的完成标准；\"加强测试\"被拆解为\"在 CI 中加入法规有效期校验脚本（owner: 张工，deadline: 2 周）\"；失败样本导入回归集，CI 每次发布必须跑回归测试。"
      outcome: "下一次法规数据更新时回归测试自动捕获问题，在上线前 2 天发现并修复；问题重演率从每季度一次降至零。"
    counterExample:
      context: "某团队事故复盘会变成追责会议，工程师担心被批评而隐瞒时间线细节。"
      antiPattern: "复盘开场是\"为什么没有发现\"而不是\"我们如何更快发现\"；文化上奖励\"没有事故\"而非\"快速复盘\"。"
      consequence: "团队复盘产出的时间线不完整，根因分析浅尝辄止；同类事故在 3 个月后以不同形式再次出现。"
    pitfalls:
      - symptom: "复盘行动项无 owner 和 deadline，会议结束后无人跟进"
        fix: "行动项必须在复盘会议结束前明确 owner 和 deadline，并在下次周会检查进度"
      - symptom: "失败样本包含用户 PII，无法安全加入测试集"
        fix: "建立样本脱敏 pipeline，用合成数据替换 PII，保留语义结构后加入回归集"
      - symptom: "复盘结论未更新发布门禁，同类问题无法在下次发布时被阻止"
        fix: "每次复盘结束时检查是否需要在发布 CI 中增加检查项，将行动项与发布流程连接"
