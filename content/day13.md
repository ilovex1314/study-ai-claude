modules:
  - id: "d13-m1"
    title: "设计知识先成为可查询数据"
    concept: "design-data"
    idea: "产品类型、行业语境、风格、色彩角色、字体尺度、版式规则和 UX 约束应被组织成独立的结构化数据域，让设计建议可查询、可复现、可审计，而不是靠模型凭感觉生成。"
    whyItMatters: "设计建议如果不能复现、不能连接适用条件、不能被审查，就会变成一次性的主观输出。结构化设计数据让团队能以相同的约束重新检索，也能在需求变更时增量更新而非全部重来。"
    engineerLens: "把设计决策当作带约束的函数调用：输入（产品、受众、任务类型、密度）决定候选空间；规则层（行业约束、可访问性规则、反模式）筛选候选。设计数据与 UI 代码解耦，修改规则不需要改组件。"
    productionExample:
      context: "某在线教育平台每次改版都需要设计师主观决定配色，同样的页面三个设计师给出三套截然不同的方案，研发无法判断标准。"
      whatTheyDid: "将设计规则结构化为 5 个数据域（产品/风格/颜色/版式/字体），每个域包含推荐项和反模式；设计师从相同输入出发检索，所有推荐携带适用条件，研发可直接从输出提取 CSS 变量名。"
      outcome: "同一产品重新检索时，三个设计师的核心 token 方案相似度达 87%（颜色角色一致、间距尺度一致）；研发不再需要逐一沟通色值含义。"
    counterExample:
      context: "另一个团队每次改版直接问 AI\"给我一个好看的深色主题\"，把输出的 hex 值逐一粘贴进组件。"
      antiPattern: "没有产品语境、没有可访问性约束、没有适用条件——hex 散落在 200 个组件中，下次改版需要全局搜索替换。"
      consequence: "深色模式改版时发现 43 个组件使用了硬编码颜色，光找色值就花了 2 天，部分低对比组合直到用户报告才被发现。"
    pitfalls:
      - symptom: "检索结果每次不一样，无法复现上次的设计决定"
        fix: "把检索查询语句和使用的数据域版本写入 design-decisions.md，形成可追溯的设计历史"
      - symptom: "设计数据涵盖不了特定行业的约束（如金融、医疗）"
        fix: "在数据域中增加行业子域，为特定行业添加合规约束（如颜色用于表意时必须加图标辅助）"
  - id: "d13-m2"
    title: "多域检索把模糊需求拆成候选方案"
    concept: "token"
    idea: "设计系统生成应同时检索产品域、风格域、色彩域、版式域和字体域，生成完整候选空间，再由规则层按行业优先级和反模式过滤；单一关键词检索会产生严重偏差。"
    whyItMatters: "\"好看的教育产品\"和\"高密度工程学习平台\"会产生完全不同的设计系统。查询粒度决定候选空间质量，候选空间质量决定最终设计系统的适配度。一个模糊查询会把所有权重放在\"好看\"上，忽略\"高密度\"、\"长阅读\"和\"可访问性\"这些工程约束。"
    engineerLens: "查询语句要同时包含产品类型、目标用户、核心任务、内容密度、风格倾向和可访问性要求。例如 `interactive AI engineering learning platform editorial high-density accessible focused light` 比 `learning app` 能提取更精准的候选。"
    productionExample:
      context: "本学习工作台在设计重构时使用了两轮检索：第一轮 `educational platform` 产生了游戏化、卡通风格的候选；第二轮 `interactive AI engineering learning platform editorial accessible focused` 产生了高对比、编辑式、字体驱动的候选。"
      whatTheyDid: "基于第二轮结果，进一步用 `--domain ux` 检索可访问性风险清单，识别出固定导航遮挡和小屏溢出两个主要风险，并在 token 定义时预留了响应式间距变量。"
      outcome: "最终设计系统通过了 WCAG 2.1 AA 对比度检查（17 个颜色对），固定导航在 375px 下自动收起为文档流内横向导航，零遮挡。"
    counterExample:
      context: "另一个学习产品团队检索了 `beautiful dark app`，把第一个推荐（Cyberpunk 霓虹风格）直接应用到课程页面。"
      antiPattern: "未检索产品域和 UX 域，未考虑长阅读和内容密度；霓虹配色在白天光线下对比度不足，动效过多干扰阅读。"
      consequence: "学习者调研显示 68% 用户感觉\"眼睛疲劳\"，产品在两周内回滚到旧版本，设计重做耗时 3 周。"
    pitfalls:
      - symptom: "多域检索产出候选太多，不知道选哪个"
        fix: "先锁定产品和 UX 约束（必须满足）过滤，再在剩余候选中按风格偏好二次筛选；规则层的反模式是强制过滤条件而非建议"
      - symptom: "不同域的推荐相互矛盾（风格域推荐暗色，产品域推荐明色）"
        fix: "产品域和 UX 域的约束优先级高于风格域；当产品类型明确需要高内容密度和长阅读时，风格候选必须让位于可读性约束"
  - id: "d13-m3"
    title: "设计系统输出落成语义 token 和组件状态"
    concept: "accessibility"
    idea: "设计系统推荐结果必须被翻译成语义 token（surface、text-primary、text-secondary、accent、danger、focus、motion-duration）、间距尺度、组件状态（default、hover、focus、active、disabled、error）和页面层级，而不是直接使用原始 hex 值。"
    whyItMatters: "直接使用 hex 值会导致颜色语义消失——你无法从 #1A2B3C 知道它是背景色、文字色还是强调色；当品牌色更新时需要全局搜索替换。语义 token 让颜色角色清晰、设计意图可传达、改版只需改 token 定义文件。"
    engineerLens: "Token 层级：基础 token（原始色值，如 --blue-600: #2563EB）→ 语义 token（--accent: var(--blue-600)）→ 组件 token（--button-bg: var(--accent)）。只在组件中使用语义 token 或组件 token，不使用基础 token。间距使用 4px 基线网格（4/8/12/16/24/32/48/64px）。"
    productionExample:
      context: "某 SaaS 产品前端有 340 个组件，颜色直接写在组件 CSS 中，主品牌色从 #4F46E5 改为 #6366F1 时需要修改 127 个文件。"
      whatTheyDid: "引入三层 token 系统，定义 28 个语义 token（含 surface-default、surface-elevated、text-primary、text-secondary、accent、accent-hover、focus-ring、danger 等），所有组件改用语义 token，基础 token 集中在 tokens.css。"
      outcome: "下次品牌色调整只修改了 tokens.css 中的 3 个基础 token 定义，全平台同步生效，耗时 20 分钟（原来需要 3 天）。"
    counterExample:
      context: "另一个团队把设计系统输出直接翻译成 Tailwind 颜色类名（如 bg-blue-600 text-gray-900），分散在 500 个模板文件中。"
      antiPattern: "语义消失，相同视觉意图用了 7 种不同的 Tailwind 类组合；可访问性团队发现 12 处颜色对比不达标，但无法快速定位和修复。"
      consequence: "可访问性修复花了 6 天，逐一检查 500 个文件；3 处修复引入了新的视觉不一致。"
    pitfalls:
      - symptom: "token 太多（超过 60 个）导致不知道用哪个"
        fix: "限制语义 token 在 20-30 个，其余用组件级 token；超过这个数量说明语义层设计有问题，应重新归类"
      - symptom: "开发使用了语义 token 但视觉结果和设计稿不一致"
        fix: "在 Storybook 或设计工具中同步 token 值，建立 token 文档页面展示每个 token 的使用场景和视觉效果"
  - id: "d13-m4"
    title: "UX 验收让设计真正可用而非仅可看"
    concept: "validation"
    idea: "真实 UI 必须在所有目标断点（375/768/1024/1440px）、键盘可达路径、焦点状态可见性、固定元素不遮挡内容、颜色对比 >= 4.5:1（正文）/ 3:1（大文字和图标）和 prefers-reduced-motion 六个维度验证，桌面截图不等于验收通过。"
    whyItMatters: "设计在桌面看起来完美，在手机上可能导航遮挡课程内容；键盘用户可能无法跳转到练习区；动效可能触发前庭障碍用户的不适。UX 验收把\"看起来好\"变成\"真实可用\"。"
    engineerLens: "建立 UX 验收 matrix：每个断点 × 每个交互状态（键盘/触摸/鼠标） × 每个可访问性维度。验收结果写入 ux-audit.md，失败项有对应修复 issue。把 axe-core 或 lighthouse accessibility 检查集成进 CI，把断点截图测试集成进 Playwright。"
    productionExample:
      context: "本学习工作台第一版右侧导航在 375px 下固定定位，遮挡了 40% 的课程正文内容；焦点环在深色背景下不可见；图示区在小屏下溢出容器。"
      whatTheyDid: "逐项修复：375px 导航改为文档流内横向标签（不固定）；焦点环改用 outline: 3px solid var(--focus-ring)（2px offset），在深色背景下使用 light-color focus ring；图示区加 overflow-x: auto 和 min-width: 320px；prefers-reduced-motion 下所有 transition 改为 0ms。把 5 项检查写入 Playwright 测试。"
      outcome: "Lighthouse 可访问性分数从 61 提升至 94；屏幕阅读器用户测试显示所有核心路径可独立完成；375px 下无内容遮挡（截图对比验证）。"
    counterExample:
      context: "某工具产品发布前只在 MacBook Pro 15 寸屏幕上做了目视检查，认为设计\"完美\"。"
      antiPattern: "没有键盘测试、没有小屏测试、没有屏幕阅读器测试，发布后收到大量无障碍投诉。"
      consequence: "因无障碍问题收到客户投诉，紧急修复花了 2 周，修复期间产品负面评价影响了转化率 (-8%)。"
    pitfalls:
      - symptom: "每次发版都要手动检查几十个 UX 指标，耗时且容易遗漏"
        fix: "把可自动化的检查（对比度、heading 层级、图片 alt、焦点顺序）集成进 CI；手动验收只检查自动化无法覆盖的交互场景"
      - symptom: "375px 断点测试时发现大量问题，修复成本很高"
        fix: "在设计稿阶段就定义移动优先的 token 和组件状态，不在实现完成后才做断点测试"
