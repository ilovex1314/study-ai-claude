# 迭代：P0.5 + F5（2026-06-27）

## 目标

把 `../study-ai` 干净重做为本地优先 AI 工程学习工作台：单一事实源内容管道、深内容契约、关系正确的图引擎、可扩展题型引擎、可验证产出闭环。

## 建了什么

按 16 个 TDD 任务推进（subagent-driven-development，但因是单人绿地项目，多数任务直接实现而非派双子代理审查）：

- 脚手架 + CI（Vite/React 19/TS/Vitest）。
- 内容 schema（zod，深度 + 图不变量）、生成器（frontmatter → 类型化）、校验器门禁。
- curriculum 域；assessment（评分器注册表 + single grader、加权评分 + 复盘）。
- learner-state 适配器（统一 key、练习证据）；practice（Rubric + 能力档案聚合）。
- 图引擎（关系正确、SVG 连线）；题型渲染注册表 + SingleQuestion。
- 课程页 UI（Hero/导航/概念卡/决策/测验/复盘）；练习证据 + 自评 Rubric + 能力档案 UI。
- 20 天深内容；产品/流程/CHANGELOG/README/本迭代文档。

最终：27 个测试全绿，`pnpm validate:content && pnpm test && pnpm build` 通过。

## 关键决策

- **frontmatter 装结构化数据**：题目/权重/边无歧义可测。
- **题型判别联合 + 双注册表**：F1–F3 只需叠加 grader + renderer + 内容。
- **问答题（F3）= 自评题**：无后端，复用练习 Rubric 机制；评分进 `selfScore`。
- **生成物不入库**：`lessons.generated.ts` 由 prebuild/CI 生成、gitignore。
- **不兜底**：缺参考/权重不为 100 直接报错。

## 与计划的偏差（实施中发现并处理）

- **Node 22 原生剥离 TS 类型**：`.mjs` 脚本可直接 `import schema.ts`，未用计划预案中的 `vite-node` 回退。
- **计划 Task 2 的 `valid` 测试夹具过薄**（1 模块/1 层/1 参考）不满足 schema 的深度最小值（3/2/2）——深度门禁是 F5-1 的核心，故扩充夹具而非削弱 schema。
- **课程导航显示 phase + capability 而非 title**：避免标题在导航与 Hero 重复导致 `getByText` 命中多个。
- **tsconfig 生产构建排除 `*.test.*`**：计划测试用了未使用的 `describe` 导入与 `as const` 只读元组，与严格 `tsc` 构建冲突；测试仍由 Vitest 运行。新增 `src/test-setup.ts` 接入 jest-dom。
- **20 天内容由 4 个并行 authoring 子代理起草，均在收尾前触发会话额度上限**，各自漏写最后一天（day10/15/20），由主代理补齐这三天（恰是 feedback/gate/flywheel 三种较复杂图型）。子代理产出的 frontmatter 有未加引号标量导致 9 个文件 YAML 解析失败，用一次性归一化脚本（JSON.stringify 重新加引号）修复，再过 schema + 语义校验。

## 验证结果

- `pnpm validate:content` → 20 lessons OK（权重和 100、图关系匹配、参考完整、恰好 20 天）。
- `pnpm test` → 27 passed。
- `pnpm build` → 通过（仅有 bundle 体积 > 500kB 的告警，因 20 天内容内联，非错误）。
